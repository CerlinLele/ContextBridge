# Embedding Model Selection

Last checked: 2026-05-04

## 1. Recommendation

For ContextBridge ISA, start with `text-embedding-3-small` as the MVP baseline embedding model.

The project goal is not to find the strongest embedding model on a public leaderboard. The goal is to build a reliable RAG assistant that helps developers understand payroll, superannuation, HR technology, compliance rules, and system integration context from trusted documents.

The MVP should prioritize:

```text
retrieval quality
source-grounded answers
citation quality
cost control
simple deployment
repeatable evaluation
```

`text-embedding-3-small` is a good default because it is cost-effective, widely supported, easy to use with LlamaIndex, and strong enough for a controlled document Q&A MVP.

## 2. Project Constraints

ContextBridge ISA should answer questions from trusted documents and show source references where possible.

The main retrieval workload is expected to include:

```text
product documentation
business rules
compliance explanations
technical notes
API or data model references
developer onboarding material
```

This means embedding model selection should be evaluated against developer-facing retrieval quality, not only generic benchmark scores.

The project should avoid comparing too many embedding models during the MVP. Each extra embedding model creates extra engineering work:

```text
separate vector index
separate embedding dimension
separate ingestion run
separate cost profile
separate latency profile
separate runtime dependency
separate evaluation result
```

## 3. Shortlist

| Use case | Model | Dimension | Reason |
| --- | --- | ---: | --- |
| MVP baseline | `text-embedding-3-small` | 1536 | Low cost, simple integration, good default retrieval quality |
| Higher retrieval quality comparison | `text-embedding-3-large` | 3072 | Stronger quality, but higher cost and larger vector storage |
| AWS-native comparison | `amazon.titan-embed-text-v2:0` | 1024 / 512 / 256 | Uses Bedrock, IAM, and AWS-native operational controls |
| Mixed text/image or complex PDF content | `cohere.embed-v4.0` | 256-1536 | Useful when documents contain images, screenshots, or visually rich PDFs |
| Local or self-hosted option | `BAAI/bge-m3` | 1024 | Useful when data should not leave the controlled runtime environment |

## 4. Recommended MVP Profiles

Use only a small number of model profiles at the beginning:

```text
dev-openai-small-v1        # baseline
dev-bedrock-titan-v2-1024  # AWS-native comparison
dev-openai-large-v1        # quality comparison
```

Add more only if there is a clear retrieval problem that the baseline cannot solve.

For example, add Cohere only if the knowledge base contains visually rich PDFs or mixed text/image content:

```text
dev-cohere-embed-v4-1536
```

Add a local model only if privacy, offline deployment, or self-hosting becomes a real requirement:

```text
dev-bge-m3-local-1024
```

## 5. Default Profile

Recommended default for the first RAG baseline:

```json
{
  "embedding_profile_id": "dev-openai-small-v1",
  "provider": "openai",
  "model": "text-embedding-3-small",
  "dimension": 1536
}
```

AWS-native comparison profile:

```json
{
  "embedding_profile_id": "dev-bedrock-titan-v2-1024",
  "provider": "bedrock",
  "model": "amazon.titan-embed-text-v2:0",
  "dimension": 1024,
  "region": "ap-southeast-2"
}
```

Confirm the selected Bedrock model is available in the target AWS Region before using it.

## 6. Evaluation Method

Do not choose the final embedding model by model name alone.

Use the same:

```text
document set
chunking strategy
metadata schema
vector database
top_k setting
test question set
answer scoring rubric
```

Evaluate at least:

```text
Recall@5
Recall@10
citation hit quality
answer groundedness
missing-context detection
latency
embedding cost
vector storage size
```

A reasonable MVP evaluation set should include 20-30 questions covering:

```text
STP concept explanation
SuperStream workflow explanation
payslip data model design
payroll validation requirement clarification
missing context detection
API or workflow impact explanation
```

## 7. Selection Rules

Use `text-embedding-3-small` unless there is evidence that retrieval quality is not good enough.

Upgrade to `text-embedding-3-large` when:

```text
the baseline misses important chunks
domain-specific concepts are frequently confused
retrieval quality is the main bottleneck
the extra cost and storage are acceptable
```

Use `amazon.titan-embed-text-v2:0` when:

```text
AWS-native operation matters more than provider flexibility
IAM-based access control is preferred
Bedrock is already approved for the project
the selected model is available in the target AWS Region
```

Use `cohere.embed-v4.0` when:

```text
documents include images, screenshots, or visually rich PDFs
mixed text/image retrieval is required
the team accepts an additional external model provider
```

Use `BAAI/bge-m3` or another local model when:

```text
data cannot leave the runtime environment
self-hosting is required
the team can operate model inference infrastructure
```

## 8. Vector Index Rule

The vector index is tied to the embedding model and dimension.

Store these fields with every vector index:

```text
embedding_profile_id
embedding_provider
embedding_model
embedding_dimension
parser_profile_id
chunk_size
chunk_overlap
vector_store
index_name
created_at
```

Changing only the generation LLM usually does not require rebuilding the vector index.

Changing the embedding model or embedding dimension requires a new or rebuilt vector index.

The query embedding model must match the embedding model used to build the target vector index.
