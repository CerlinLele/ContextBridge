# PDF Parsing

## Purpose

This document defines the PDF parsing strategy for ContextBridge RAG ingestion.

PDF should not be treated as a single uniform source type. Different PDFs require different parser profiles depending on document structure, content type, and downstream retrieval needs.

Examples:

```text
pdf / sectioned policy document       -> section-aware text parser
pdf / structured file specification   -> table-aware specification parser
pdf / screenshot-heavy workflow guide -> OCR and image-aware parser
pdf / scanned document                -> OCR-first parser
pdf / report with financial tables    -> table-aware report parser
```

The parser should be selected from metadata such as:

```text
source_format
mime_type
source_type
structure_type
business_domain
standard_family
parser_profile
```

## Recommended PDF Parser Profiles

Initial parser profiles:

```text
pdf-sectioned-document
pdf-table-aware-specification
pdf-screenshot-workflow-guide
pdf-ocr-scanned-document
pdf-table-aware-report
```

For the first implementation pass, focus on:

```text
pdf-table-aware-specification
```

This is the most important profile for SAFF sources because the key knowledge is often stored in dense field requirement tables.

## Tooling Recommendation

### Primary Local Stack

Use PyMuPDF plus pdfplumber.

Recommended split:

- PyMuPDF for page metadata, text blocks, coordinates, images, and table detection.
- pdfplumber for debugging table extraction and visual inspection of table boundaries.
- pypdf for lightweight metadata checks, page count checks, encryption checks, and attachment checks.

This gives enough local control for a deterministic parser while keeping the dependency footprint reasonable.

### PyMuPDF

Use as the main PDF inspection and extraction library.

Useful for:

- page size and orientation
- text blocks with coordinates
- image detection
- table detection
- extracting page-level metadata for downstream chunks

Official documentation:

```text
https://pymupdf.readthedocs.io/
```

### pdfplumber

Use for table extraction experiments and table boundary debugging.

Useful for:

- inspecting characters, lines, rectangles, and layout objects
- extracting tables from Word-generated PDFs
- visual debugging while tuning parser settings

Official project:

```text
https://github.com/jsvine/pdfplumber
```

### pypdf

Use only as a lightweight utility layer.

Useful for:

- metadata extraction
- page count
- encryption detection
- simple text extraction smoke tests
- attachment checks

Do not rely on pypdf alone for field specification tables.

Official documentation:

```text
https://pypdf.readthedocs.io/
```

### Docling

Use as an optional higher-level local parser when quick Markdown or structured document conversion is useful.

Useful for:

- PDF-to-structured-document conversion
- table structure extraction
- OCR-enabled local parsing
- RAG-oriented document conversion

Official documentation:

```text
https://docling-project.github.io/docling/
```

### Unstructured

Use as an optional general document partitioning framework when the parser must support many document classes.

Useful for:

- partitioning PDFs into elements
- OCR fallback
- high-resolution strategies for complex layouts
- unifying parsing behavior across document formats

This dependency stack is heavier than PyMuPDF plus pdfplumber, so it is better as a later option rather than the first local parser loop.

Official documentation:

```text
https://docs.unstructured.io/
```

### LlamaParse

Use as a cloud-based benchmark or fallback, not as the only parser decision.

Useful for:

- quickly evaluating extraction quality
- complex table and image-heavy PDFs
- generating LLM-ready Markdown or structured outputs

This can be useful to compare against the local parser output, but the production decision should account for privacy, cost, reproducibility, and vendor dependency.

Official documentation:

```text
https://developers.llamaindex.ai/
```

## Recommended V1 Parser Flow

```text
PDF source
-> inspect metadata and page characteristics
-> classify source by PDF parser profile
-> extract text, tables, images, and page metadata according to profile
-> normalize extracted content into structured artifacts
-> keep raw extracted text for auditability
-> optionally OCR pages or regions when embedded text is missing
-> write parsed artifacts
-> generate RAG nodes from structured rows and section chunks
```

## SAFF Field Specification Example: GESB

```text
knowledge_base/sources/structured-business-file-formats/saff/specifications/gesb/superstream-payroll-data-specification.pdf
```

Document title:

```text
GESB SuperStream Alternative File Format (SAFF) Payroll Data Specification
```

### File Characteristics

This PDF should be treated as a structured business file format specification, not as a normal narrative PDF.

Observed characteristics:

- 34 pages.
- Approximately 990 KB.
- Not encrypted.
- Generated from Microsoft Word for Microsoft 365.
- Contains selectable embedded text on most pages.
- Uses both portrait pages and landscape pages.
- Contains document metadata for title, author, subject, creation date, modification date, and keywords.
- Does not contain an embedded spreadsheet attachment.
- Pages 1-9 contain front matter, change history, references, table of contents, and explanatory sections.
- Pages 10-27 contain the core SAFF field specification tables.
- Pages 28-34 contain sample SAFF file sections, mostly represented as embedded images or screenshots.

Important metadata observed:

```text
Author: GESB
Title: GESB SuperStream Alternative File Format (SAFF) Payroll Data Specification
Subject: GESB SuperStream Alternative File Format (SAFF) Payroll Data Specification
Keywords: GESB SuperStream Alternative File Format (SAFF) Payroll Data Specification
Version in document: 1.8
Date updated in document: 02/04/2026
Creator: Microsoft Word for Microsoft 365
Producer: Microsoft Word for Microsoft 365
```

### Parsing Implications

Do not parse this document with a generic PDF-to-text flow and then sentence-split the result.

The main value of this source is the field-level specification. The parser should preserve the structure of each field row and section.

Recommended parser profile:

```text
parser_profile = "pdf-table-aware-specification"
structure_type = "field-specification"
source_type = "structured-business-file-format"
format_name = "saff"
business_domain = "payroll-superannuation"
standard_family = "superstream"
```

### Recommended Output Model

The table-heavy section should produce structured rows. A useful normalized output shape is:

```text
source_id
source_path
document_title
document_version
document_date
page_number
section_number
section_title
column_number
field_name
description
requirements_text
mandatory
required_by_gesb
data_type
length
format
values
notes
mig_reference
des_reference
raw_text
parser_profile
parser_version
content_hash
```

The parser should keep the raw row text alongside normalized fields. This makes it possible to debug parsing errors and re-normalize fields later without re-reading the PDF.

### Recommended Handling by Page Range

#### Pages 1-9

Use normal text extraction with heading detection.

Preserve:

- document purpose
- acceptable contribution types
- delivery instructions
- payment details
- negative contribution guidance
- SMSF guidance
- SAFF structure summary
- page numbers
- section numbers

These pages can become section-level RAG chunks.

#### Pages 10-27

Use table-aware extraction.

These pages contain the core SAFF data requirements for GESB accumulation funds. They should be parsed into field-level rows rather than page-level text chunks.

The parser should identify repeated table headers such as:

```text
Column
Field Name
Description
Requirements
Required by GESB?
MIG 2.0 Reference
DES Spec 5.8 Reference
```

The parser should also preserve each section heading, for example:

```text
11.4.2. Section: Header
11.4.3. Section: Sender details
11.4.4. Section: Payer details
11.4.5. Section: Payee/Receiver details
11.4.6. Section: Employer details
11.4.7. Section: Super fund member common details
11.4.8. Section: Super fund member contributions details
```

#### Pages 28-34

Treat these pages as lower priority unless sample file screenshots are required.

These pages contain sample SAFF file sections, but the PDF text extraction returns little usable text. The content appears to be represented mostly as embedded images.

For SAFF structure, prefer the official ATO spreadsheet already stored in the knowledge base:

```text
knowledge_base/sources/structured-business-file-formats/saff/official/ato/superstream-alternative-file-format-v1.0.xlsx
```

Use OCR on pages 28-34 only if screenshot content is needed for a specific retrieval or validation task.

### RAG Chunking Recommendation

Use different chunk types for different parts of the document.

Recommended chunk types:

```text
section_summary
field_specification_row
table_section_summary
sample_file_screenshot_note
```

For field rows, one row should usually become one searchable node. Do not merge unrelated SAFF columns into one large chunk unless the table section is being summarized separately.

Recommended metadata for each field row node:

```text
source_id
source_path
page_number
section_number
section_title
column_number
field_name
required_by_gesb
mandatory
data_type
parser_profile
parser_version
content_hash
```

This supports precise retrieval questions such as:

```text
What does GESB require for column 59?
Which SAFF fields are required by GESB?
What is the required format for Pay Period End Date?
Which fields are not used by GESB?
```

### Recommendation

Use the GESB PDF as a fund-specific SAFF requirements source.

Use the ATO spreadsheet as the stronger primary source for general SAFF structure.

For local parser implementation, start with:

```text
pymupdf
pdfplumber
pypdf
```

Then add Docling, Unstructured, OCR, or LlamaParse only after the deterministic local parser has been evaluated against this document.
