# Parser Local Development Runtime

## Question

If we want to run the parser locally, do we need to start local Docker services for the vector database, DynamoDB, S3, SQS, and other cloud dependencies?

Also, should the local Dockerfile be different from the cloud Dockerfile?

## Recommendation

Do not start with the full cloud-like runtime locally.

Build the parser in layers. Start with the smallest local loop, then add local infrastructure only when the parser core is working.

The parser Docker image should be as close as possible to the cloud image. Local and cloud behavior should differ mostly through runtime configuration, not separate application code or completely different Dockerfiles.

## Local Development Layers

### Layer 1: Minimal Parser Loop

Start with local files and no external services:

```text
local files
-> parser
-> LlamaIndex Documents
-> Nodes
-> print or save parsed artifacts
```

This layer does not require:

- vector database
- DynamoDB
- S3
- SQS
- LocalStack
- ECS

Use this layer to validate:

- PDF parsing
- Excel parsing
- table extraction
- page metadata
- sheet metadata
- chunk boundaries
- source metadata preservation

This is the fastest development loop and should be the first target.

### Layer 2: Local Vector Search

After parsing and chunking work locally, add a local vector store:

```text
local files
-> parser
-> chunking
-> embeddings
-> local vector store
-> query test
```

Recommended local vector store options:

- Chroma
- Qdrant in Docker

For this layer, DynamoDB can still be skipped. Ingestion status can be written to a local JSON file, SQLite, or an in-memory store while the parser behavior is still evolving.

### Layer 3: Local Ingestion Status

After vector indexing works, add ingestion status tracking:

```text
local files
-> parser
-> vector store
-> local status store
```

Development options:

- JSON file
- SQLite
- DynamoDB Local

Use DynamoDB Local when the status schema is stable enough that testing DynamoDB-specific behavior is useful.

### Layer 4: Local AWS-Like Runtime

Only add local AWS replacements when the parser worker needs to exercise S3 and SQS behavior:

```text
LocalStack S3
-> LocalStack SQS
-> parser worker
-> local vector store
-> DynamoDB Local
```

Useful local services:

- LocalStack for S3 and SQS
- DynamoDB Local for ingestion status tables
- Qdrant or local OpenSearch for vector search

This layer is useful for testing:

- S3 object references
- SQS message handling
- retry behavior
- dead-letter behavior
- worker startup and shutdown
- status transitions
- local end-to-end ingestion

Do not start here. This layer is slower and has more moving parts.

## Dockerfile Guidance

Prefer one parser Dockerfile:

```text
Dockerfile
```

Use it to build the parser worker image for both local and cloud environments.

The local environment should be controlled through:

```text
docker-compose.local.yml
.env.local
environment variables
local service endpoints
```

The cloud environment should be controlled through:

```text
ECS task definition
AWS environment variables
AWS Secrets Manager
IAM task role
CloudWatch logs
```

Avoid maintaining two unrelated Dockerfiles such as:

```text
Dockerfile.local
Dockerfile.production
```

Separate Dockerfiles usually drift over time and can hide deployment bugs.

## Acceptable Dockerfile Differences

Small build-time differences are acceptable when needed, for example:

- optional development tools
- test dependencies
- local debugging tools
- different entrypoint for interactive debugging

Prefer using Docker build targets instead of separate Dockerfiles:

```text
Dockerfile
  base
  runtime
  dev
```

Example:

```text
docker build --target dev -t contextbridge-parser-worker:dev .
docker build --target runtime -t contextbridge-parser-worker:runtime .
```

The parser application code should remain the same.

## Configuration Boundary

Local and cloud should differ through configuration values:

```text
APP_ENV=local
AWS_REGION=ap-southeast-2
SOURCE_BUCKET=contextbridge-kb-sources-local
ARTIFACT_BUCKET=contextbridge-kb-artifacts-local
INGESTION_QUEUE_URL=http://localstack:4566/...
VECTOR_INDEX=contextbridge-kb-local
MODEL_PROFILE_ID=local-embedding
PARSER_PROFILE_ID=saff-v1
```

For local services, also configure endpoints:

```text
S3_ENDPOINT_URL=http://localstack:4566
SQS_ENDPOINT_URL=http://localstack:4566
DYNAMODB_ENDPOINT_URL=http://dynamodb-local:8000
VECTOR_STORE_URL=http://qdrant:6333
```

Cloud deployment should omit local endpoint overrides and use AWS service endpoints by default.

## Suggested Development Order

1. Local files to parser output.
2. Local files to LlamaIndex Documents and Nodes.
3. Preserve required metadata on each node.
4. Add local vector store indexing.
5. Add local query testing.
6. Add local ingestion status using JSON or SQLite.
7. Replace local status store with DynamoDB Local.
8. Add LocalStack S3 input.
9. Add LocalStack SQS worker polling.
10. Deploy the same parser image to ECS Fargate.

## Decision

Do not require the full local Docker runtime at the beginning.

Start with a minimal local parser loop, then add a local vector database, then add ingestion status, and only later add LocalStack and DynamoDB Local for cloud-like integration testing.

Use one main Dockerfile for the parser worker wherever possible. Keep local and cloud differences in environment variables, Docker Compose, ECS task definitions, and service endpoints.
