# Parser Code Structure

## Question

If we want to start writing the parser implementation, do we need a `src` folder?

## Recommendation

Yes. The `src` folder is not technically required by Python, but it is the better default for ContextBridge.

The parser is expected to become a deployable ECS Fargate worker, not just a one-off script. It will need to handle SQS messages, load source files from S3, run the LlamaIndex parsing and chunking pipeline, generate embeddings, write to the vector store, persist parsed artifacts, and update DynamoDB ingestion status.

Keeping this implementation under `src` makes the code easier to test, package, import, and copy into a Docker image.

## Recommended Initial Structure

```text
ContextBridge/
  docs/
  infra/
  knowledge_base/
  src/
    contextbridge_parser/
      __init__.py
      main.py
      config.py
      pipeline.py
      readers/
        __init__.py
      parsers/
        __init__.py
      embeddings/
        __init__.py
      vectorstores/
        __init__.py
      status_store/
        __init__.py
  tests/
  pyproject.toml
  Dockerfile
```

## File Responsibilities

`src/contextbridge_parser/main.py`

Worker entry point. This should start with a local development mode, then later support polling SQS or handling a single SQS message.

`src/contextbridge_parser/config.py`

Runtime configuration loaded from environment variables, such as:

```text
APP_ENV
AWS_REGION
SOURCE_BUCKET
ARTIFACT_BUCKET
INGESTION_QUEUE_URL
VECTOR_INDEX
MODEL_PROFILE_ID
PARSER_PROFILE_ID
```

`src/contextbridge_parser/pipeline.py`

Core ingestion flow shared by both local development and AWS deployment:

```text
reader
-> parser
-> LlamaIndex Documents
-> Nodes
-> embeddings
-> vector store
-> artifacts/status updates
```

`src/contextbridge_parser/readers/`

Source readers. Start with a local folder reader for development, then add an S3 reader for AWS.

`src/contextbridge_parser/parsers/`

Parser selection and file-specific parsing behavior for PDF, Excel, and future document types.

`src/contextbridge_parser/embeddings/`

Embedding provider integration, such as Bedrock embeddings or a local development provider.

`src/contextbridge_parser/vectorstores/`

Vector store integration, such as OpenSearch Serverless, pgvector, Chroma, or Qdrant.

`src/contextbridge_parser/status_store/`

Ingestion lifecycle persistence. In production this should update DynamoDB tables such as `rag_sources` and `rag_ingestion_runs`.

## Scripts Versus Source Code

It is acceptable to create a quick local script while experimenting:

```text
scripts/parse_local.py
```

However, scripts should only call reusable code from `src/contextbridge_parser`. Core parsing, chunking, embedding, and persistence logic should not live permanently inside `scripts`.

Good pattern:

```text
scripts/parse_local.py
-> imports contextbridge_parser.pipeline
```

Avoid:

```text
scripts/parse_local.py
-> contains all parser, embedding, and vector store logic
```

## Suggested Development Order

1. Create `src/contextbridge_parser`.
2. Add `pyproject.toml` with package metadata and dependencies.
3. Implement local parsing first using files from `knowledge_base/sources`.
4. Add a local vector store such as Chroma or Qdrant.
5. Add tests for parser output and metadata preservation.
6. Add S3 reader support.
7. Add SQS worker behavior.
8. Add production vector store and DynamoDB status updates.
9. Add Dockerfile for the ECS parser worker.

## Decision

Use a `src` folder for ContextBridge parser code.

Start with local development support, but keep the interfaces aligned with the AWS worker design so the same parser pipeline can later run inside ECS Fargate.
