# AWS Parser Deployment with LlamaIndex

## Context

The current SAFF knowledge source contains structured business file format documents:

```text
knowledge_base/sources/structured-business-file-formats/saff/
  official/ato/superstream-alternative-file-format-v1.0.xlsx
  operational-guides/mercer/saff-support-guide-employer-portal.pdf
  specifications/csc-pssap/pssap-saff-extended-data-requirements.pdf
  specifications/gesb/superstream-payroll-data-specification.pdf
```

This is not an ideal request-time parsing workload. The parser should run as an asynchronous ingestion pipeline that reads source files, extracts text and tables, chunks content, generates embeddings, and writes searchable nodes into the vector store.

## Recommendation

Deploy the parser as an ECS Fargate worker using LlamaIndex.

Recommended V1 architecture:

```text
S3 raw source bucket
-> S3 ObjectCreated event
-> SQS ingestion queue
-> ECS Fargate parser worker
-> LlamaIndex reader/parser pipeline
-> embedding model
-> OpenSearch Serverless or pgvector
-> DynamoDB ingestion status
-> S3 parsed artifacts
```

## Why ECS Fargate Instead of Lambda

Lambda can work for small files and short parsing jobs, but it is a poor default for production document ingestion when the parser may need PDF extraction, Excel parsing, OCR, table handling, or embedding batches.

Key Lambda constraints:

- Maximum invocation duration is 15 minutes.
- Container images can become large when parser and OCR dependencies are included.
- Temporary disk and memory need careful sizing for large PDF and Excel files.
- Parsing and embedding should not block user-facing API requests.

Use Lambda only for lightweight orchestration, such as receiving S3 events and starting an ECS task. For the parser itself, use ECS Fargate.

Use AWS Batch later if ingestion becomes high-volume, long-running, or heavily scheduled.

## Runtime Responsibilities

The parser worker should:

1. Receive an S3 object reference from SQS.
2. Load the source file from S3.
3. Select the right parser based on file type and parser profile.
4. Extract text, tables, page metadata, and sheet metadata.
5. Convert extracted content into LlamaIndex Documents.
6. Split Documents into Nodes.
7. Generate embeddings using the configured embedding profile.
8. Upsert nodes into the vector store.
9. Write parsed artifacts and failure reports to S3.
10. Update source and ingestion status in DynamoDB.

```text
  contextbridge-parser-worker-dev

  rag_sources
  rag_ingestion_runs

Vector store
  OpenSearch Serverless vector index
  or Aurora PostgreSQL / RDS PostgreSQL with pgvector
```

Separate the source and artifact buckets. If an S3 event writes parsed outputs back into the same bucket and prefix that triggered ingestion, it can create an event loop.

## Example Worker Shape

```python
from llama_index.core.ingestion import IngestionPipeline
from llama_index.core.node_parser import SentenceSplitter
from llama_index.readers.s3 import S3Reader


def parse_s3_source(bucket: str, key: str):
    reader = S3Reader(bucket=bucket, key=key)
    documents = reader.load_data()

    pipeline = IngestionPipeline(
        transformations=[
            SentenceSplitter(chunk_size=800, chunk_overlap=120),
            # Add embedding transformation here.
        ],
        # Add vector_store here.
    )

    nodes = pipeline.run(documents=documents)
    return nodes
```

The production implementation should wrap this with:

- SQS message handling
- source registry lookup
- content hashing
- parser profile selection
- embedding profile selection
- retry and dead-letter handling
- structured logging
- DynamoDB status updates

## Runtime Configuration

Use environment variables for non-secret runtime configuration:

```text
APP_ENV=dev
AWS_REGION=ap-southeast-2
SOURCE_BUCKET=contextbridge-kb-sources-dev
ARTIFACT_BUCKET=contextbridge-kb-artifacts-dev
INGESTION_QUEUE_URL=...

MODEL_PROFILE_ID=dev-bedrock-titan-claude
PARSER_PROFILE_ID=saff-v1
```

Use AWS Secrets Manager for provider API keys. Do not store secret values in model profile JSON files or task definitions.

## DynamoDB Source Registry

```text

content_hash

ingestion_status
parser_profile_id
embedding_profile_id
vector_index_name

```

The vector store should optimize retrieval. DynamoDB should track ingestion lifecycle, audit state, and operational status.

## Deployment Flow

1. Build the parser Docker image.
2. Push the image to ECR.
3. Create the ECS Fargate task definition.
4. Create the SQS ingestion queue and dead-letter queue.
5. Configure S3 ObjectCreated notifications for the raw source prefix.
6. Run the worker as an ECS service that polls SQS, or trigger one-off ECS tasks from events.
7. Write parsed artifacts to the artifact bucket.
8. Upsert chunks and embeddings to the vector store.
9. Update DynamoDB ingestion status.
10. Add CloudWatch logs, metrics, and alarms.

## Local Development Path

For local development, keep the same parser interface but use local inputs and a local vector store:

```text
local folder reader
-> LlamaIndex parser pipeline
-> local embedding provider or configured cloud embedding provider
-> Chroma or Qdrant
```

This keeps local testing fast while allowing the AWS implementation to use the same parsing and chunking logic.

## Decision

For ContextBridge V1:

```text
Use S3 + SQS + ECS Fargate + LlamaIndex + OpenSearch Serverless + DynamoDB.
```

Use Lambda only for orchestration if needed.

Use AWS Batch later if ingestion volume or runtime requirements outgrow an always-on ECS worker or event-triggered ECS tasks.
