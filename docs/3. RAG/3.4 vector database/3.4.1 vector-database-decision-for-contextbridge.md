# Vector Database Decision for ContextBridge

## Recommendation

For ContextBridge ISA, use:

```text
Primary AWS vector database: Amazon OpenSearch Serverless
Local MVP / prototype option: Qdrant or Chroma
Fallback if PostgreSQL becomes a core app database: Aurora PostgreSQL / RDS PostgreSQL + pgvector
```

The main recommendation is still `Amazon OpenSearch Serverless`, even though the project has metadata.

## Why Metadata Does Not Force pgvector

ContextBridge needs metadata and vectors to be searchable together.

Each vector record should store both the embedding vector and retrieval metadata, for example:

```text
node_id
embedding_vector
chunk_text
source_id
source_uri
title
page_number
section
chunk_index
document_type
category
parser_profile_id
embedding_profile_id
created_at
```

This does not mean the project must use PostgreSQL or pgvector.

OpenSearch Serverless can also store vector fields and metadata fields in the same index, and query them together through:

```text
vector similarity search + metadata filtering
```

For ContextBridge, the actual requirement is:

```text
metadata filtering + vector search + citation retrieval
```

OpenSearch Serverless fits this requirement well.

## Recommended Storage Split

Use the vector database for retrieval-time metadata.

Use the source registry for ingestion, lifecycle, and audit metadata.

### OpenSearch Vector Record

Store fields needed during retrieval and citation:

```text
node_id
vector
chunk_text
source_id
title
source_uri
page_number
section
category
document_type
embedding_profile_id
parser_profile_id
```

Example retrieval filters:

```json
{
  "category": "saff",
  "document_type": "pdf",
  "embedding_profile_id": "dev-openai-small-v1"
}
```

### DynamoDB Source Registry

Store fields needed for source management:

```text
source_id
s3_bucket
s3_key
content_hash
status
ingestion_status
parser_profile_id
embedding_profile_id
vector_index_name
chunk_count
ingested_at
last_modified_at
failure_reason
```

This keeps retrieval fast while keeping ingestion state and operational metadata easy to manage.

## Decision Matrix

| Option | Fit for ContextBridge | Use when |
| --- | --- | --- |
| Amazon OpenSearch Serverless | Best default AWS choice | The project needs RAG search, metadata filtering, citations, and possible hybrid search later |
| Aurora/RDS PostgreSQL + pgvector | Conditional fit | PostgreSQL is already the main app database and SQL joins over app data are important |
| Qdrant | Good custom RAG option | The team wants a dedicated vector database with simple local/dev ergonomics |
| Chroma | Good local prototype option | The team wants the fastest local RAG loop before AWS deployment |
| S3 Vectors | Not the first choice | Cost-efficient vector storage matters more than retrieval tuning and advanced filtering |

## Final Decision

For the AWS implementation of ContextBridge ISA:

```text
Choose Amazon OpenSearch Serverless.
```

For the local MVP:

```text
Use Qdrant or Chroma behind the same internal vector store adapter.
```

Keep the application dependent only on a small internal interface:

```text
upsert(nodes, embeddings, metadata)
query(vector, top_k, filters)
delete_by_source(source_id)
```

This allows the project to start quickly in local development and still move cleanly to OpenSearch Serverless for the AWS version.
