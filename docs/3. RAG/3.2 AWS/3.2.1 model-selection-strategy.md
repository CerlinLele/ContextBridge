# Model Selection Strategy

## 1. Goal

The RAG system should support selectable model profiles instead of hard-coding a single model.

Model choices should be separated into:

```text
embedding model
generation LLM
optional reranker model
parser mode
```

This allows each environment to choose different models while keeping ingestion, retrieval, and answer generation logic stable.

## 2. Model Selection Scope

The model selection scope should cover four independent choices:

```text
1. Embedding model
2. Generation LLM
3. Optional reranker
4. Parser mode
```

Each choice should be stored in configuration and referenced through profile IDs.

## 3. Embedding Models

The embedding model is the most important model choice for vector search.

It determines:

```text
embedding provider
embedding model ID
embedding dimension
vector index dimension
compatible query embedding model
```

Changing the embedding model or embedding dimension usually requires rebuilding the vector index.

## 4. Recommended Embedding Model Options

Recommended V1 embedding options:

```text
Bedrock Titan Text Embeddings V2
Bedrock Cohere Embed English v3
Bedrock Cohere Embed Multilingual v3
OpenAI text-embedding-3-small
OpenAI text-embedding-3-large
local / self-hosted embedding model
```

For an AWS-first deployment in Australia, start by checking support in:

```text
ap-southeast-2
```

Practical candidates to evaluate:

```text
amazon.titan-embed-text-v2:0
cohere.embed-english-v3
cohere.embed-multilingual-v3
```

Typical dimensions:

```text
amazon.titan-embed-text-v2:0  -> 256 / 512 / 1024
cohere.embed-english-v3       -> 1024
cohere.embed-multilingual-v3  -> 1024
OpenAI text-embedding-3-small -> provider-defined, store explicitly
OpenAI text-embedding-3-large -> provider-defined, store explicitly
```

Do not assume model availability across all AWS Regions. Confirm the selected model in the target Region before deployment.

Reference:

- Amazon Bedrock Knowledge Bases supported models and Regions: https://docs.aws.amazon.com/en_us/bedrock/latest/userguide/knowledge-base-supported.html
- OpenAI data residency and endpoint model support: https://developers.openai.com/api/docs/guides/your-data#which-models-and-features-are-eligible-for-data-residency

## 5. Generation LLMs

The generation LLM is used to produce the final answer from retrieved context.

It can be selected independently from the embedding model.

Recommended V1 LLM provider options:

```text
Bedrock Anthropic Claude
Bedrock Amazon Nova
Bedrock Mistral
Bedrock Meta
Bedrock Cohere
OpenAI GPT models
local / private hosted LLM
```

For an AWS-first deployment in Australia, evaluate models available in:

```text
ap-southeast-2
```

Practical candidates to check:

```text
Amazon Nova Lite
Amazon Nova Micro
Amazon Nova Pro
Anthropic Claude Haiku
Anthropic Claude Sonnet
Mistral Large
```

The exact model IDs should be environment-specific and confirmed against current provider documentation before deployment.

Reference:

- Amazon Bedrock supported foundation models: https://docs.aws.amazon.com/bedrock/latest/userguide/models-supported.html
- OpenAI data residency and endpoint model support: https://developers.openai.com/api/docs/guides/your-data#which-models-and-features-are-eligible-for-data-residency

## 6. Optional Reranker Models

A reranker is optional for V1.

It can improve retrieval quality by reordering candidate chunks after vector search.

Recommended reranker options:

```text
Bedrock Amazon Rerank
Bedrock Cohere Rerank
external Cohere Rerank API
local reranker, such as bge-reranker
none
```

If the application must keep model inference in a specific AWS Region, confirm reranker Region support before enabling it.

If reranker support is not available in the target Region, start with:

```text
reranker: none
```

Reference:

- Amazon Bedrock reranking supported Regions and models: https://docs.aws.amazon.com/bedrock/latest/userguide/rerank-supported.html

## 7. Parser Modes

Parser mode is not an LLM choice, but it directly affects retrieval quality.

It should be managed as part of the model/profile strategy.

Recommended parser options:

```text
SentenceSplitter
TokenTextSplitter
HierarchicalNodeParser
SemanticSplitterNodeParser
custom parser
optional vision parser for PDFs or images
```

Recommended V1 default:

```text
framework: llamaindex
node_parser: SentenceSplitter
chunk_size: 800
chunk_overlap: 120
```

Changing parser mode, chunk size, or chunk overlap should trigger evaluation and may require reindexing.

## 8. Recommended V1 Profiles

Suggested starting profiles:

```text
dev:
  embedding: Bedrock Titan Text Embeddings V2
  llm: Amazon Nova Lite or Claude Haiku/Sonnet
  reranker: none
  parser: SentenceSplitter

staging:
  embedding: same candidate family as prod
  llm: Claude Sonnet or Nova Pro
  reranker: optional
  parser: SentenceSplitter

prod:
  embedding: Titan V2 or Cohere Multilingual v3
  llm: Claude Sonnet or Nova Pro
  reranker: decided by Region support and evaluation
  parser: fixed after evaluation
```

These are starting points, not permanent choices.

Production defaults should be selected based on:

```text
retrieval quality
answer groundedness
citation accuracy
latency
cost
Region support
data residency requirements
```

## 9. Model Profile Shape

Example profile:

```json
{
  "profile_id": "dev-bedrock-titan-nova-lite",
  "environment": "dev",
  "embedding": {
    "provider": "bedrock",
    "model_id": "amazon.titan-embed-text-v2:0",
    "dimension": 1024,
    "region": "ap-southeast-2"
  },
  "llm": {
    "provider": "bedrock",
    "model_id": "amazon.nova-lite-v1:0",
    "region": "ap-southeast-2",
    "temperature": 0.2,
    "max_tokens": 1200
  },
  "reranker": {
    "provider": "none",
    "model_id": null,
    "region": null
  },
  "parser": {
    "framework": "llamaindex",
    "node_parser": "SentenceSplitter",
    "chunk_size": 800,
    "chunk_overlap": 120
  },
  "retrieval": {
    "top_k": 5
  }
}
```

Model IDs should be checked before deployment and should not be hard-coded into ingestion or query business logic.

## 10. Required Metadata

Store these values with every vector index:

```text
environment
embedding_provider
embedding_model
embedding_dimension
embedding_profile_id
parser_profile_id
parser_version
chunk_size
chunk_overlap
vector_store
index_name
region
created_at
```

Store these values with every query or answer trace:

```text
query_id
model_profile_id
embedding_provider
embedding_model
llm_provider
llm_model
reranker_provider
reranker_model
prompt_version
retrieval_profile_id
top_k
latency
token_usage
answer_source_count
created_at
```

## 11. Compatibility Rules

The query embedding model must match the embedding model used by the target vector index.

Rules:

```text
1. Do not query a vector index with a different embedding model.
2. Do not query a vector index with a different embedding dimension.
3. Rebuild the vector index when embedding model or dimension changes.
4. Re-run evaluation when LLM, prompt, parser, chunking, reranker, or retrieval settings change.
5. Store exact provider, model ID, dimension, and Region for each profile.
6. Validate Region support before deploying an environment.
```

## 12. Recommended Decision

For V1, start with:

```text
embedding:
  Bedrock Titan Text Embeddings V2

llm:
  Amazon Nova Lite for dev
  Claude Sonnet or Nova Pro for staging/prod evaluation

reranker:
  none

parser:
  LlamaIndex SentenceSplitter
```

Then compare alternatives through a fixed evaluation set before changing staging or production defaults.
