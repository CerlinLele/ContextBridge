# Model Selection Strategy

## 1. Goal

The RAG system should not limit model selection to a fixed provider list.

Model selection should follow the LlamaIndex configuration model:

```text
LLM component
embedding component
node parser / text splitter component
optional reranker / postprocessor component
```

The application should load these components from configuration and pass them into LlamaIndex instead of hard-coding Bedrock, OpenAI, or any other provider in business logic.

## 2. Design Principle

The system should be provider-agnostic.

Supported model providers can include:

```text
Bedrock
OpenAI
Cohere
Hugging Face
Ollama
local models
self-hosted models
private hosted models
custom LlamaIndex-compatible implementations
```

This document should not define a closed list of allowed models.

Instead, each environment should define model profiles that describe how to instantiate LlamaIndex-compatible components.

## 3. LlamaIndex Configuration Model

LlamaIndex supports global defaults through `Settings`:

```python
from llama_index.core import Settings

Settings.llm = llm
Settings.embed_model = embed_model
Settings.text_splitter = text_splitter
```

LlamaIndex also supports local overrides when creating indexes and query engines:

```python
index = VectorStoreIndex.from_documents(
    documents,
    embed_model=embed_model,
    transformations=transformations,
)

query_engine = index.as_query_engine(llm=llm)
```

Recommended project rule:

```text
Use explicit local components for ingestion and query paths.
Use Settings only as a controlled default inside the application boundary.
```

This avoids accidental provider changes when multiple profiles exist.

References:

- LlamaIndex Settings: https://docs.llamaindex.ai/en/stable/module_guides/supporting_modules/settings/
- LlamaIndex Embeddings: https://docs.llamaindex.ai/en/stable/module_guides/models/embeddings/
- LlamaIndex custom LLMs: https://docs.llamaindex.ai/en/stable/module_guides/models/llms/usage_custom/

## 4. Profile Types

Separate these profile types:

```text
embedding_profile
llm_profile
parser_profile
reranker_profile
retrieval_profile
model_profile
```

`model_profile` can compose the other profiles:

```text
model_profile
  -> embedding_profile
  -> llm_profile
  -> parser_profile
  -> reranker_profile
  -> retrieval_profile
```

This keeps each choice independently changeable and traceable.

## 5. Embedding Profile

The embedding profile defines the LlamaIndex embedding component used for indexing and query embedding.

Example:

```json
{
  "embedding_profile_id": "dev-openai-small-v1",
  "component_type": "embedding",
  "llamaindex_class": "llama_index.embeddings.openai.OpenAIEmbedding",
  "provider": "openai",
  "model": "text-embedding-3-small",
  "dimension": 1536,
  "config": {
    "embed_batch_size": 100
  }
}
```

Another example using a local Hugging Face model:

```json
{
  "embedding_profile_id": "dev-local-bge-v1",
  "component_type": "embedding",
  "llamaindex_class": "llama_index.embeddings.huggingface.HuggingFaceEmbedding",
  "provider": "huggingface",
  "model": "BAAI/bge-base-en-v1.5",
  "dimension": 768,
  "config": {}
}
```

The exact `dimension` must be stored explicitly.

The vector index must be created for the same embedding dimension.

## 6. LLM Profile

The LLM profile defines the LlamaIndex LLM component used for answer generation.

Example:

```json
{
  "llm_profile_id": "dev-openai-gpt-v1",
  "component_type": "llm",
  "llamaindex_class": "llama_index.llms.openai.OpenAI",
  "provider": "openai",
  "model": "gpt-4.1-mini",
  "config": {
    "temperature": 0.2,
    "max_tokens": 1200
  }
}
```

Example using a local model through Ollama:

```json
{
  "llm_profile_id": "dev-local-ollama-v1",
  "component_type": "llm",
  "llamaindex_class": "llama_index.llms.ollama.Ollama",
  "provider": "ollama",
  "model": "llama3.1",
  "config": {
    "request_timeout": 360.0,
    "context_window": 8000
  }
}
```

The LLM can change without rebuilding the vector index, but evaluation should be rerun when the LLM changes.

## 7. Parser Profile

The parser profile defines how documents are split into nodes.

Example:

```json
{
  "parser_profile_id": "sentence-800-120-v1",
  "component_type": "parser",
  "llamaindex_class": "llama_index.core.node_parser.SentenceSplitter",
  "framework": "llamaindex",
  "config": {
    "chunk_size": 800,
    "chunk_overlap": 120
  },
  "version": 1
}
```

Parser options can include:

```text
SentenceSplitter
TokenTextSplitter
HierarchicalNodeParser
SemanticSplitterNodeParser
custom parser
vision-aware parser for PDFs or images
```

Changing parser settings can require reindexing because generated nodes may change.

## 8. Reranker Profile

The reranker profile is optional.

It can be implemented as a LlamaIndex node postprocessor or a custom retrieval step.

Example:

```json
{
  "reranker_profile_id": "none",
  "component_type": "reranker",
  "enabled": false,
  "llamaindex_class": null,
  "provider": "none",
  "model": null,
  "config": {}
}
```

Example with a provider-backed reranker:

```json
{
  "reranker_profile_id": "cohere-rerank-v1",
  "component_type": "reranker",
  "enabled": true,
  "llamaindex_class": "llama_index.postprocessor.cohere_rerank.CohereRerank",
  "provider": "cohere",
  "model": "rerank-english-v3.0",
  "config": {
    "top_n": 5
  }
}
```

The system should allow `reranker: none`.

## 9. Retrieval Profile

The retrieval profile stores search behavior independent from model provider.

Example:

```json
{
  "retrieval_profile_id": "top-k-5-v1",
  "top_k": 5,
  "similarity_cutoff": null,
  "filters_enabled": true,
  "hybrid_search": false
}
```

Changing retrieval settings does not usually require reindexing, but it does require evaluation.

## 10. Composed Model Profile

A composed model profile references the selected component profiles.

Example:

```json
{
  "model_profile_id": "dev-openai-local-parser-v1",
  "environment": "dev",
  "embedding_profile_id": "dev-openai-small-v1",
  "llm_profile_id": "dev-openai-gpt-v1",
  "parser_profile_id": "sentence-800-120-v1",
  "reranker_profile_id": "none",
  "retrieval_profile_id": "top-k-5-v1",
  "status": "active",
  "is_default": true
}
```

This keeps the application flexible without limiting providers.

## 11. Component Factory

The application should have an internal factory that maps profile config to LlamaIndex objects.

Example sketch:

```python
from llama_index.core import VectorStoreIndex

embedding = component_factory.create_embedding(embedding_profile)
llm = component_factory.create_llm(llm_profile)
transformations = component_factory.create_transformations(parser_profile)
postprocessors = component_factory.create_postprocessors(reranker_profile)

index = VectorStoreIndex.from_documents(
    documents,
    embed_model=embedding,
    transformations=transformations,
)

query_engine = index.as_query_engine(
    llm=llm,
    similarity_top_k=retrieval_profile.top_k,
    node_postprocessors=postprocessors,
)
```

The factory should validate that required integration packages and credentials are available.

## 12. Configuration Storage

Store profile configuration in environment-specific files first:

```text
infra/configs/dev/model-profiles.json
infra/configs/staging/model-profiles.json
infra/configs/prod/model-profiles.json
```

Later, these can move into DynamoDB:

```text
rag_model_profiles
rag_embedding_profiles
rag_llm_profiles
rag_parser_profiles
rag_reranker_profiles
rag_retrieval_profiles
```

Secrets should not be stored in these profiles.

Store secret references only:

```json
{
  "provider": "openai",
  "secret_ref": "/contextbridge/dev/openai/api-key"
}
```

## 13. Required Metadata

Store these values with every vector index:

```text
environment
embedding_profile_id
embedding_provider
embedding_model
embedding_dimension
parser_profile_id
parser_version
chunk_size
chunk_overlap
vector_store
index_name
created_at
```

Store these values with every query or answer trace:

```text
query_id
model_profile_id
embedding_profile_id
llm_profile_id
parser_profile_id
reranker_profile_id
retrieval_profile_id
embedding_provider
embedding_model
llm_provider
llm_model
prompt_version
top_k
latency
token_usage
answer_source_count
created_at
```

## 14. Compatibility Rules

The system should avoid provider restrictions, but it still needs compatibility rules.

Rules:

```text
1. The query embedding component must match the embedding component used by the target vector index.
2. The vector index dimension must match the embedding dimension.
3. Rebuild the vector index when embedding model or dimension changes.
4. Rebuild or reingest when parser output changes.
5. Re-run evaluation when LLM, prompt, parser, chunking, reranker, or retrieval settings change.
6. Validate that the selected LlamaIndex integration package is installed.
7. Validate that required credentials or secret references exist before running ingestion or query.
```

## 15. Recommended V1 Decision

For V1, do not restrict model selection to Bedrock.

Use this approach:

```text
model selection:
  LlamaIndex profile-driven components

provider policy:
  open by default in dev
  explicitly configured per environment

embedding:
  any LlamaIndex-compatible embedding component

llm:
  any LlamaIndex-compatible LLM component

parser:
  LlamaIndex SentenceSplitter as the first default

reranker:
  none by default, pluggable later
```

The only hard requirement is that the selected components must be reproducible from configuration and traceable in ingestion and query metadata.
